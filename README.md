# artifact-branch-publish-action

A composite action that appends artifact files generated by E2E tests and the like (screenshots, videos, HTML reports, etc.) as commits to a dedicated branch in the repository, and returns commit-SHA-pinned `github.com/…/blob/…?raw=true` URLs. Since these share the same origin as `github.com` itself, they're viewable by any signed-in viewer with repository access — even on private repositories.

> [!NOTE]
> Browser behavior differs by file type. Images, audio, and video are served with the correct MIME type and display/play inline, but **HTML files are served as `text/plain` and shown as source rather than rendered**. See [File types and browser behavior](#file-types-and-browser-behavior-important).

Useful when you want to embed images or videos generated during a run directly into a workflow's Step Summary or a comment. Artifacts accumulate on a dedicated, rotating branch (`<branch-prefix>-gen-N`) and never affect the current checkout or the rest of the repository's history.

## Usage

The calling workflow needs **`permissions: contents: write`** (it pushes to the dedicated branch).

```yaml
permissions:
  contents: write

steps:
  - uses: actions/checkout@v7

  # E2E tests etc. generate the artifact files here
  - run: npm test

  - id: publish
    uses: aki77/artifact-branch-publish-action@v1
    with:
      files: artifacts/**/*.gif

  - run: |
      echo "${{ steps.publish.outputs.urls }}" | while IFS= read -r url; do
        echo "![screenshot]($url)" >> "$GITHUB_STEP_SUMMARY"
      done
```

## inputs

| Name | Required | Default | Description |
| --- | --- | --- | --- |
| `files` | ✅ | - | Glob of files to publish (space-separated for multiple). Example: `artifacts/**/*.gif` |
| `branch-prefix` | | `artifacts` | Prefix for the generation branches. The branch actually used is `<branch-prefix>-gen-N` |
| `retain-days` | | `30` | The **minimum lifetime guarantee** for the returned URL. Once the active generation's oldest commit becomes older than this many days, the next push rotates to a new generation |
| `github-token` | | `${{ github.token }}` | Token used for pushing. Requires `contents: write` permission |

## outputs

| Name | Description |
| --- | --- |
| `urls` | Newline-separated full viewable URLs, one per file: `https://github.com/<repo>/blob/<sha>/<run_id>-<run_attempt>/<path>?raw=true`. `<run_id>-<run_attempt>` is a subdirectory automatically numbered per invocation (run) |
| `commit-sha` | The SHA of the pushed commit |
| `branch` | The name of the generation branch actually pushed to (e.g. `artifacts-gen-7`) |

## Embedding in a Step Summary

`urls` is a newline-separated list of full, ready-to-use URLs — one per published file — so you can loop over it directly (the loop runs once for a single file, so the same snippet works whether you published one file or many):

```yaml
- run: |
    echo "${{ steps.publish.outputs.urls }}" | while IFS= read -r url; do
      echo "![screenshot]($url)" >> "$GITHUB_STEP_SUMMARY"
    done
```

Since each URL includes a subdirectory unique to each run (`<run_id>-<run_attempt>`), every invocation (run) is guaranteed to be isolated into its own separate directory.

## How it works (generation rotation)

Internally, artifacts are managed using a "two-branch generation rotation" scheme.

- The actual branch name is `<branch-prefix>-gen-N` (e.g. `artifacts-gen-7`).
- Commits are **appended only** to the branch of the current generation. Once that generation's **oldest commit becomes older than `retain-days`**, the next push starts a new generation (`gen-N+1`) from a clean state (an orphan commit).
- After a successful push, only the **two most recent generations** are ever kept on the remote; anything older is deleted.
- Since each branch is append-only, the content behind a previously returned commit SHA never changes (it's immutable).

## URL lifetime (important)

- Because the URL is pinned to a commit SHA, it always points to an immutable snapshot. You never need to worry about filename collisions or overwrites.
- Lifetime guarantee: valid until the generation branch containing that commit is deleted. It is guaranteed to remain reachable for **at least `retain-days` days, and at most roughly `2 × retain-days` days** (since the two newest generations are always kept, and a generation only becomes eligible for rotation/deletion once it exceeds `retain-days`).
- However, generation rotation and old-generation deletion are only evaluated **when a push happens**. In a repository where publishing doesn't happen for a long time, an old generation may remain undeleted during that period (i.e., the lifetime skews longer). Conversely, this is not a guarantee that it disappears at exactly `retain-days` — it is only a **lower bound of at least `retain-days`**.
- Beyond that, the entire generation branch is deleted and the URL becomes unreachable (404).
- Regarding storage reduction: deleting an old generation branch makes its commits unreachable, but actual disk space reduction is left to GitHub's automatic GC, which may run with some delay and cannot be forced from this action. Note that nothing is left behind locally (in the runner) either — the action operates inside a temporary directory and never touches the caller's `.git`.

## File types and browser behavior (important)

Files are served from `github.com` via `?raw=true`, and **how a browser handles the URL depends on the file's type**. This is GitHub's own behavior and cannot be overridden by this action (the `Content-Type` and `Content-Disposition` response headers are controlled entirely by GitHub):

| File type | Served as | In the browser |
| --- | --- | --- |
| Images (`.png`, `.jpg`, `.gif`, `.svg`, `.webp`, …) | correct MIME (e.g. `image/png`) | **displayed inline** — works great for embedding in Step Summaries / comments |
| Audio / video (`.mp3`, `.mp4`, …) | often correct MIME (e.g. `audio/mpeg`), but GitHub adds `Content-Disposition: attachment` | **always downloaded**, never played inline in the browser |
| **HTML** (`.html`, `.htm`) | `text/plain` | **shown as source, not rendered** (see below) |
| Text / Markdown (`.txt`, `.md`, …) | `text/plain` | shown as plain text |

### HTML is not rendered

GitHub serves **HTML as `text/plain` with `X-Content-Type-Options: nosniff`**, so opening a published `.html` URL shows its **source instead of a rendered page**. This is a deliberate GitHub anti-XSS measure and applies to `github.com` via `?raw=true` (used by this action), `raw.githubusercontent.com`, the Contents API, and Git LFS alike — there is no URL, query-string, or header trick that makes GitHub render (or force-download) a single HTML file. It cannot be worked around from this action.

If you need a **rendered** HTML report, host it somewhere that serves it with `Content-Type: text/html`:

- **Public repositories:** [GitHub Pages](https://docs.github.com/en/pages), or a preview proxy such as [`htmlpreview.github.io`](https://htmlpreview.github.io/).
- **Private repositories:** an external host you control (Cloudflare Pages + Access, Netlify, S3 + CloudFront, …), or access-controlled GitHub Pages (**GitHub Enterprise Cloud only**). Deploy the report there in a separate workflow step and surface that URL yourself.

This action remains a great fit for **images, audio, video, and downloadable artifacts** — just not for in-browser HTML rendering.

## Notes

- Since the URLs share the same origin as `github.com` itself, on private repositories they're viewable by any **signed-in viewer with repository access**. Unauthenticated viewers get a 404.
- Artifacts are automatically isolated into a subdirectory unique to each run (`<run_id>-<run_attempt>`), so file contents never collide across different runs. Push conflicts from concurrent runs (a force-push race on the generation branch) are resolved via automatic retry.
- This action never affects the caller's checked-out working tree or git state. It operates entirely within its own temporary clone.

## `comment-files` sub-action

A sub-action that publishes files matching a glob (internally using `artifact-branch-publish` above), writes a message plus the file list as markdown to the Step Summary, and — only when running in a PR context — posts the same content as a PR comment. Image files (`.png`, `.jpg`, `.jpeg`, `.gif`, `.webp`, `.svg`, `.avif`, `.bmp`, `.ico`) are embedded inline as `![alt](url)`; every other file is listed as a plain `- [name](url)` bullet link, with the bullet list placed before the inline images so it stays visible without scrolling past them. If the glob matches zero files, it exits successfully without publishing, writing the Step Summary, or commenting.

The calling workflow needs **`permissions: contents: write`** and **`permissions: pull-requests: write`**.

```yaml
name: Example

on:
  pull_request:

permissions:
  contents: write
  pull-requests: write

jobs:
  comment-files:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v7

      # E2E tests etc. generate artifact files here
      - run: npm test

      - uses: aki77/artifact-branch-publish-action/comment-files@main
        with:
          files: artifacts/**/*
          message: 'Artifacts from this run:'
```

### inputs

| Name | Required | Default | Description |
| --- | --- | --- | --- |
| `files` | ✅ | - | Glob of files to publish and comment (space-separated for multiple, e.g. `artifacts/**/*`) |
| `message` | | `''` | Optional leading message placed above the file list in the comment body. |
| `marker` | | `<!-- artifact-branch-comment-files -->` | Hidden HTML marker appended to the comment. Used to find and replace prior comments from this action. |
| `branch-prefix` | | `artifacts` | Prefix for the dedicated rotating branches. Passed through to artifact-branch-publish. |
| `retain-days` | | `30` | Minimum number of days a published URL is guaranteed to stay valid. Passed through to artifact-branch-publish. |
| `github-token` | | `${{ github.token }}` | Token used to push and to post the comment (needs contents: write and pull-requests: write). |

### outputs

| Name | Description |
| --- | --- |
| `urls` | Newline-separated full viewable URLs, one per file. Mirrors `artifact-branch-publish`'s `urls` output. |
| `commit-sha` | The commit SHA the artifacts were pushed as. |
| `branch` | The actual generation branch the artifacts were pushed to. |
| `matched` | Whether the files pattern matched at least one file (true/false). |
| `comment-posted` | Whether a PR comment was posted (true/false). |

### Upsert behavior

PR comments are a pseudo-upsert: before posting, this action uses [`aki77/delete-pr-comments-action`](https://github.com/aki77/delete-pr-comments-action) to delete any prior comment whose body contains `marker` (a hidden HTML comment), then posts a fresh comment. This keeps the PR to a single comment from this action even if the workflow runs multiple times on the same PR.

### Non-PR runs

When the workflow isn't running in a PR context (e.g. `push`, `workflow_dispatch`, or any event where a PR number can't be resolved), the PR comment step is skipped and only the Step Summary is written.

### Notes

- Since the URLs share the same origin as `github.com` itself, on private repositories images embedded via these URLs render for any **signed-in viewer with repository access**; unauthenticated viewers get a 404. Non-image files are posted as regular links, which authenticated viewers with repository access can still open.
- Non-image files (including HTML) are posted as plain links, and opening an HTML link shows its source rather than a rendered page. See [File types and browser behavior](#file-types-and-browser-behavior-important) for why, and what to do if you need a rendered HTML report.
