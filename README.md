# artifact-branch-publish-action

A composite action that appends artifact files generated by E2E tests and the like (screenshots, videos, HTML reports, etc.) as commits to a dedicated branch in the repository, and returns a commit-SHA-pinned URL prefix viewable via `raw.githubusercontent.com`.

Useful when you want to embed images or videos generated during a run directly into a workflow's Step Summary or a comment. Artifacts accumulate on a dedicated, rotating branch (`<branch-prefix>-gen-N`) and never affect the current checkout or the rest of the repository's history.

## Usage

The calling workflow needs **`permissions: contents: write`** (it pushes to the dedicated branch).

```yaml
permissions:
  contents: write

steps:
  - uses: actions/checkout@v4

  # E2E tests etc. generate the artifact files here
  - run: npm test

  - id: publish
    uses: aki77/artifact-branch-publish-action@v1
    with:
      files: artifacts/**/*.gif

  - run: |
      echo "![screenshot](${{ steps.publish.outputs.url-prefix }}artifacts/screenshot.gif)" >> "$GITHUB_STEP_SUMMARY"
```

## inputs

| Name | Required | Default | Description |
| --- | --- | --- | --- |
| `files` | ✅ | - | Glob of files to publish (space-separated for multiple). Example: `artifacts/**/*.gif` |
| `branch-prefix` | | `artifacts` | Prefix for the generation branches. The branch actually used is `<branch-prefix>-gen-N` |
| `retain-days` | | `30` | The **minimum lifetime guarantee** for the returned URL. Once the active generation's oldest commit becomes older than this many days, the next push rotates to a new generation |
| `github-token` | | `${{ github.token }}` | Token used for pushing. Requires `contents: write` permission |

## outputs

| Name | Description |
| --- | --- |
| `url-prefix` | Commit-SHA-pinned base URL: `https://raw.githubusercontent.com/<repo>/<sha>/<run_id>-<run_attempt>/`. Has a trailing slash. `<run_id>-<run_attempt>` is a subdirectory automatically numbered per invocation (run) |
| `commit-sha` | The SHA of the pushed commit |
| `branch` | The name of the generation branch actually pushed to (e.g. `artifacts-gen-7`) |

## Embedding in a Step Summary

Since `url-prefix` has a trailing slash, appending the relative path you passed to `files` gives you the viewable URL.

```yaml
- run: |
    echo "![screenshot](${{ steps.publish.outputs.url-prefix }}artifacts/screenshot.gif)" >> "$GITHUB_STEP_SUMMARY"
```

If you specify `artifacts/**/*.gif` for `files`, artifacts are placed keeping their relative path structure (e.g. `artifacts/screenshot.gif`). Just append that relative path after `url-prefix`. Also, since `url-prefix` includes a subdirectory unique to each run (`<run_id>-<run_attempt>`), every invocation (run) is guaranteed to be isolated into its own separate directory.

## How it works (generation rotation)

Internally, artifacts are managed using a "two-branch generation rotation" scheme.

- The actual branch name is `<branch-prefix>-gen-N` (e.g. `artifacts-gen-7`).
- Commits are **appended only** to the branch of the current generation. Once that generation's **oldest commit becomes older than `retain-days`**, the next push starts a new generation (`gen-N+1`) from a clean state (an orphan commit).
- After a successful push, only the **two most recent generations** are ever kept on the remote; anything older is deleted.
- Since each branch is append-only, the content behind a previously returned commit SHA never changes (it's immutable).

## URL lifetime (important)

- Because the URL is pinned to a commit SHA, it always points to an immutable snapshot. You never need to worry about filename collisions or overwrites.
- Lifetime guarantee: valid until the generation branch containing that commit is deleted. It is guaranteed to remain reachable for **at least `retain-days` days, and at most roughly `2 × retain-days` days** (since the two newest generations are always kept, and a generation only becomes eligible for rotation/deletion once it exceeds `retain-days`).
- However, generation rotation and old-generation deletion are only evaluated **when a push happens**. In a repository where publishing doesn't happen for a long time, an old generation may remain undeleted during that period (i.e., the lifetime skews longer). Conversely, this is not a guarantee that it disappears at exactly `retain-days` — it is only a **lower bound of at least `retain-days`**.
- Beyond that, the entire generation branch is deleted and the URL becomes unreachable (404).
- Regarding storage reduction: deleting an old generation branch makes its commits unreachable, but actual disk space reduction is left to GitHub's automatic GC, which may run with some delay and cannot be forced from this action. Note that nothing is left behind locally (in the runner) either — the action operates inside a temporary directory and never touches the caller's `.git`.

## Notes

- On private repositories, `raw.githubusercontent.com` URLs cannot be viewed without authentication. Viewers also need access to the repository.
- Artifacts are automatically isolated into a subdirectory unique to each run (`<run_id>-<run_attempt>`), so file contents never collide across different runs. Push conflicts from concurrent runs (a force-push race on the generation branch) are resolved via automatic retry.
- This action never affects the caller's checked-out working tree or git state. It operates entirely within its own temporary clone.
